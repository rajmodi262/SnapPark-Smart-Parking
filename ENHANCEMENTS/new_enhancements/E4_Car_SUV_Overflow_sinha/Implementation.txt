STEP 1: Add areCarSlotsFull() Method (Service Layer)
───────────────────────────────────────────────────
File: ParkingService.java

Added method: areCarSlotsFull()
- Uses Java Stream API to filter all slots
- Counts total CAR-type slots
- Counts AVAILABLE CAR-type slots
- Returns true only when availableCarSlots == 0

Logic:
  List<ParkingSlot> allSlots = slotDAO.findAll();

  long totalCarSlots = allSlots.stream()
      .filter(s -> s.getType() == VehicleType.CAR)
      .count();

  long availableCarSlots = allSlots.stream()
      .filter(s -> s.getType() == VehicleType.CAR
                && s.getStatus() == SlotStatus.AVAILABLE)
      .count();

  return totalCarSlots > 0 && availableCarSlots == 0;

This mirrors the existing areBikeSlotsFull() method pattern.


STEP 2: Add Overflow Detection to Check-in API
──────────────────────────────────────────────
File: WebServer.java → handleCheckin()

Added detection block AFTER the shared slot logic:

  boolean isOverflowBooking = false;

  if (vehicleType == CAR && targetSlot.type == SUV) {
      if (!parkingService.areCarSlotsFull()) {
          sendResponse(400, "Car slots are still available.");
          return;  // Block — must use car slot first
      }
      isOverflowBooking = true;
  }

This ensures:
  ✓ Only triggers when a CAR selects an SUV slot
  ✓ Validates all car slots are full before allowing
  ✓ If car slots have space, rejects with helpful error
  ✓ Sets flag for JSON response

After this block, the normal booking flow continues:
  lockSlot → confirmBooking → startSession → respond

No SHARED status needed (just AVAILABLE → OCCUPIED).
